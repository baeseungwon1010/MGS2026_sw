# 2주차

계층 구조

상위 계층 서비스에 관심 가지고, 하위 계층 서비스를 사용

인터넷 프로토콜 스택

| 5 | application |
| --- | --- |
| 4 | transport |
| 3 | network |
| 2 | link |
| 1 | physical |

physical(물리 계층): bit 단위, 프레임 내부의 각 비트를 다음 노드로 이동

link(데이터 링크 계층): frame 단위, 전체 프레임을 한 네트워크에서 이웃 네트워크 요소로 이동

각 서비스는 해당 링크 계층에서 사용하는 프로토콜에 의해 결정, 

 network(네트워크 계층=ip 계층): datagram 단위, 한 호스트에서 다른 호스트로 최적 경로 선택, 데이터그램 전송(라우팅), ip 프로토콜 수행

transport(전송 계층):  segment 단위, 클라이언트와 서버간 메시지를 전송하는 서비스 제공

TCP:  연결 지향형 서비스 제공, 전달 보장, 흐름 제어, 분할, 혼잡 제어 기능

UDP: 비연결형, 신뢰성/흐름제어/혼잡제어 제공 안함

application 계층: message 단위, 하위 계층에 의존해 전달, http/smtp/ftp 등

캡슐화: 각 계층에 따른 헤더를 붙여 전송

segment(transport header+message)

packet(network header+segment)

frame(ip header+packet)

link 계층

노드: 링크 계층 프로토콜 실행 장치(호스트, 라우터, 스위치, ap 등)

링크: 통신 경로상 인접한 노드들을 연결하는 통신 채널

subnet: lan을 구분하는 범위

framing(프레임화): frame으로 캡슐화

link acess(링크 접속): mac 프로토콜 링크상으로 frame 전송 규칙 명시

신뢰적 전달: 응답 재전송 통한 서비스 제공, 오류 발생한 링크에서 정정

오류 검출/정정: 오류 있는 데이터그램 전달 필요 없어 오류 검출 진행

스위치는 링크 계층에서 동작: 링크 계층 frame 교환, ip 계층 주소 체계 사용 x(각 계층별 독립적 주소 기법 필요성)

mac 주소: 호스트나 라우터의 네트워크 인터페이스 가짐, 6byte 구성, frame안 목적 mac 검사 후 전달

arp(address resolution protocol): 네트워크 계층 주소와 링크 계층 주소 사이 변환 프로토콜 같은 lan 내부에서만 동

ip→mac 주소 반환(매핑), ttl(time-to-live, 매핑이 언제 사라지는지), plug and play(자동)

같은 lan 내부 case: ip를 브로드캐스트하여 같은 lan 내부 전체에게 질의(arp)

다른 lan case: subnet이 다른걸 판단, 같은 lan에 없으므로 게이트웨이(연결 통로)에 넘겨 게이트웨이가 해당 subnet을 라우팅 테이블에서 검색→해당 subnet을 가지는 lan에 브로드캐스트하여 연결된 전체에게 질의(arp)

ethernet: lan 기술, 비연결형/비신뢰적, tranport에서 신뢰성/연결성 구현 가능, 재전송/뉴 구분 못함

ethernet frame: data:ip 데이터그램 운반(크면 단편화), 목적지/출발지 mac(6byte), 타입 필드:ip등의 네트워크 계층 프로토콜 종류

스위치: link 계층 frame 전달, 전송 속도/수용 속도 차이로 인한 패킷 손실을 막기 위한 버퍼 존재, mac 주소 사용 frame 전달, 2계층

필터링: frame을 전달/폐기 결정, 포워딩: 프레임이 전송될 인터페이스 결정→ 전송

네트워크 계층

호스트 간 transport 계층 segment를 전달

데이터 평면: datagram 전달/ 제어 평면: 전달시 라우팅 알고리즘, 포워딩

제공 가능 서비스 :  보장 전달(출발 패킷→목적지 도착 보장), 지연 제한 시간 보장, 순서화 패킷 전달(송신 순서대로 도착), 최소 대역폭 보장(특정 속도 이하 전송하면 전달됨을 보장), 보안(segment에 대해 기밀성 제공)

최선형 서비스 제공(순서 보장 x, 목적지 보장 x, 지연 보장 x, 최소 대역폭 보장 x): 단순할 수록 실시간에 유리

ip 프로토콜

모든 호스트/라우터는 ip datagram 수신할 수 있도록 ip 주소 가짐

클래스 주소체계: 필요한 호스트 ip 개수에 따라 네트워크 크기 할당

브로드캐스트 주소: 모든 호스트에 전달하는 주소

NAT(network address translation): 하나 공인 ip 주소 사용, 프라이빗 네트워크 내 여러 개 ip 주소와 통신, 가상 아이피 체제

DHCP(Dynamic Host Configuration Protocol): 네트워크에 연결된 장치들에게 자동으로 ip 주소 관련 설정 할당

트랜스포츠 계층

논리적 통신 제공(호스트들이 직접 연결된 것 처럼 보임)

전송계층 프로토콜: tcp, udp, quic 등

다중화: 호스트의 데이터 모으고 각 세그멘터에 헤더 정보 추가/캡슐화하여 네트워크 계층 전달

역다중화: 올바른 소켓 전달 작업

UDP: 정교한 제어(최소한, 직접 구현), 연결 설정 부재(handshake 없음, 연결 상태 유지 x), 혼잡제어 부재(필요한 작업 못하는 상태 시 대처 어려움)

TCP: 신뢰성 있는 데이터 전송(수신 여부 확인, 재전송, 패킷 순서 유지), 연결 지향성(handshake), 흐름제어(송수신 측)/혼잡제어(네트워크 내) 패킷 수 조절, 중복 없음/순서 유지 보장

MSS(max segment size): 최대 데이터양, 패킷 쪼개기 가능, MTU에 의해 결정,MTU-TCP/IP 헤더

MTU: link 계층 frame의 크기 중 가장 큰 것

Seq num: 세그먼트에 잇는 첫 바이트 스트림 번호/ Ack num: 상대방이 기대하는 Seq num

손실 segment 발견 위해 타임아웃/재전송 매커니즘 사용, 타임아웃은 rtt보다 커야함

RTT(round-trip-time): 왕복시간

3way handshake(연결)

1. 클라→서버: 연결 요청(syn=1,seq=a)
2. 서버→클라; 서버 응답(syn=1,seq=b,ack=a+1)
3. 클라→서버: 연결 완료(syn=0, seq=a+1,ack=b+1)

3번째 메세지에 페이로드 포함 가능, 연결시 syn은 0으로 설정

4way handshake(종료)

1. 클라→서버: 연결 종료 요청(fin=1)
2. 서버→클라: 요청 확인(ack)
3. 서버→클라: 연결 종료 요청(fin=1)
4. 클라→서버: 요청 확인(ack)

종료는 누구나 가능, 호스트의 자원(버퍼, 메모리) 회수
